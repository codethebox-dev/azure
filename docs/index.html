Introduction

My Azure Project is an open-source infrastructure toolkit built for deploying and managing cloud-native workloads on Microsoft Azure. It provides a consistent, repeatable pattern for provisioning resources, enforcing governance policies, and automating operational runbooks through Azure-native tooling.

Whether you're migrating an existing workload or building greenfield architecture, this project gives you the scaffolding to ship production-ready infrastructure with confidence.

Overview ¬∂

At its core, My Azure Project orchestrates three layers of Azure infrastructure: networking (Virtual Networks, NSGs, Private Endpoints), compute (AKS, Container Apps, Azure Functions), and observability (Log Analytics, Azure Monitor, Application Insights). Each layer is independently configurable through a central config.yaml file.

Core Module Network Layer VNets, subnets, NSGs, private endpoints, and DNS zones. Core Module Compute Layer AKS clusters, Container Apps, and Azure Functions with autoscaling. Core Module Observability Centralized logging, metrics, alerts, and dashboards. Core Module Identity & Access Managed identities, RBAC assignments, and Key Vault integration.

How It Works ¬∂

The project uses a modular Bicep/Terraform structure. Each module is independently deployable and follows the principle of least privilege by default. The deployment pipeline integrates directly with GitHub Actions and Azure DevOps.

1

Configure your environment

Edit config.yaml to define your Azure subscription, resource groups, and desired modules.

2

Authenticate with Azure

Run az login and set your subscription context. Service principal auth is recommended for CI/CD.

3

Deploy infrastructure

Execute the deploy script. Resources are provisioned in dependency order with full rollback support.

# Clone and enter the project
git clone https://github.com/your-org/my-azure-project.git
cd my-azure-project

# Install dependencies
pip install -r requirements.txt

# Deploy to your subscription
python deploy.py --env production --config config.yaml

Architecture ¬∂

My Azure Project follows a hub-spoke topology by default. A central hub Virtual Network hosts shared services (Azure Firewall, Bastion, DNS Private Resolver), while spoke networks are provisioned per workload or team. All cross-spoke traffic is routed through the hub firewall.

‚ö†
Warning Deploying without a hub network is supported for development environments only. Production deployments must use hub-spoke topology to satisfy security requirements.
